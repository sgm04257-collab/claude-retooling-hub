강의 핵심 12가지

Anthropic Academy 공식 강의 21강의 핵심 내용을 아코디언으로 정리했습니다. 클릭해서 펼쳐보세요.

① 시작하기 — 설치 및 초기화▼
Node.js 18+ 필요.
npm install -g @anthropic-ai/claude-code cd project && /init
초기 스캔 및 스타터 메모리 파일 생성. Claude Code = AI 현장 직원. tool use로 파일 읽기·수정·명령어 실행.
② CLAUDE.md — AI 업무 매뉴얼▼
대상: Tech stack, Architecture
이유: Purpose, Design decisions
방법: Build/test commands, Workflows, Gotchas

매 세션 시작 시 자동 로드. Git에 커밋해서 팀 공유.
③ 메모리 파일 3계층▼
~/.claude/CLAUDE.md # 글로벌 (전체) ./CLAUDE.md # 프로젝트 (Git 공유) ./frontend/CLAUDE.md # 서브폴더 (스코프)
각 파일 200줄 미만 · 절대 상위 컨텍스트 덮어쓰지 않음.
④ 모범 사례▼
먼저 /init 후 출력 다듬기 · 구체적인 지시사항 · @filename 문서 참조
Claude가 유추 못하는 유의사항 추가 · 메모리는 간결하게 · 팀 공유 위해 Git 커밋
⑤ 프로젝트 파일 구조▼
your-project/ ├── CLAUDE.md ├── .claude/ │ ├── settings.json │ ├── skills/ │ │ ├── code-review/SKILL.md │ │ └── testing/SKILL.md │ ├── commands/deploy.md │ └── agents/security-reviewer.md └── src/
⑥ 스킬 추가 — 자연어 호출▼
--- name: testing patterns description: jest testing patterns — 언제 쓸지로 작성 allowed_tools: Read, Grep, Glob --- # Testing Patterns Use describe+it+AAA pattern Use factory mocks
설명 필드는 "언제 이 스킬을 실행할지"로 작성. 자동 활성화에 핵심.
⑦ AI 엔지니어 스킬 아이디어▼
① 코드 리뷰 (보안·XSS 자동 감지)
② 테스트 패턴 (TDD 자동화)
③ 커밋 메시지 (Conventional Commits)
④ 도커 배포 스킬
⑤ 코드베이스 시각화
⑥ API 디자인 가이드
⑧ Hooks — 자동 안전 가드레일▼
"hooks": { "PreToolUse": [{ "matcher": "Bash", "hooks": [{ "type": "command", "command": "scripts/sec.sh", "timeout": 5 }] }] }
종료코드 0 → 허용, 1+ → 차단
⑨ 권한 및 안전▼
"permissions": { "allow": ["Read:*","Bash:git:*","Write:*:*.md"], "deny": ["Read:env:*","Bash:sudo:*"] }
⑩ 4단계 아키텍처▼
L1 – CLAUDE.md: 지속 컨텍스트 및 규칙
L2 – Skills: 자동 호출 지식 팩
L3 – Hooks: 안전 게이트 및 자동화
L4 – Agents: 자체 컨텍스트 하위 에이전트

12가지 핵심 설정법

잘 쓰는 사람과 못 쓰는 사람의 차이는 CLAUDE.md 하나. 세션이 끊겨도 규칙이 살아있으려면 여기에 기록해야 합니다.

파일 위치 & 범위
관리 정책 (강제)
/etc/claude-code/CLAUDE.md
조직 전체 강제 적용 · 사용자 오버라이드 불가. macOS: ~/Library/Application Support/ClaudeCode/CLAUDE.md
프로젝트 (팀 공유)
./CLAUDE.md
프로젝트 아키텍처 · 코딩 표준 · Git으로 팀 공유 필수. 모노레포에서 가장 중요.
사용자 (개인)
~/.claude/CLAUDE.md
개인 코드 스타일 · 선호도. Git에 올리지 않는 개인 설정.
❌ vs ✅ 지침 작성법
❌ Claude가 무시합니다
"코드를 제대로 포맷합니다"
"변경사항을 테스트합니다"
"API를 src/api에 위치시킵니다"
✅ 검증 가능한 지시
"2칸 들여쓰기 사용"
"커밋 전 npm test 실행"
"API는 src/api/handler에 위치"
핵심 규칙 12가지
④
200줄 이하 유지
길수록 준수율 하락. 많아지면 @import나 .claude/rules/로 분리.
⑤
@import로 분리
@README @package.json @~/.claude/my-prefs.md — Git 싫은 개인 설정은 홈 디렉토리에서 로드.
⑥
.claude/rules/ 분리
code-style.md / testing.md / security.md — 주제별 분리 → 팀 유지관리 쉬워짐.
⑦
경로별 규칙
paths: "src/api/**/*.ts"
해당 파일 열 때만 규칙 로드. 컨텍스트 절약.
⑧
자동 메모리 구분
CLAUDE.md = 내가 씀
Auto Memory = Claude가 씀
"항상 pnpm 써" → 자동 기억.
⑨
로드 순서
관리정책 → 사용자 → 상위 → 프로젝트 → 하위
claudeMdExcludes로 다른 팀 파일 제외.
⑩
/memory 습관화
현재 세션 로드 파일 확인. 목록에 없으면 Claude는 못 봄. 쓰고 나서 반드시 확인.
⑪
/compact 후 소실 주의
대화에만 있던 지침은 compact 후 사라짐. CLAUDE.md에 기록해야 영구 유지.
⑫
핵심 요약
CLAUDE.md = 구체적·간결·200줄↓

.claude/rules/ = 대규모

8
장애 대응 런북
에러 증상 → 관련 로그 수집 → 보고서 자동 작성.
예: "500 에러 급증" → 원인 리포트

인프라 운영

⚠️
① 함정 목록(Pitfall List)을 반드시 넣을 것 — 가장 중요
AI가 자주 틀리는 부분을 미리 적어두면 같은 실수를 반복하지 않습니다. 새로운 실패가 발견될 때마다 한 줄씩 추가하는 것이 핵심입니다.
## Pitfalls - axios 타임아웃 기본값은 0 (무한대) — 반드시 timeout: 5000 명시 - React key는 index 대신 고유 ID 사용 - 2026-01 이후 버전은 deprecated API 사용 금지
📂
② 점진적 공개(Progressive Disclosure) — 파일로 나눌 것
상세한 API 시그니처는 references/api.md에, 출력 템플릿은 assets/ 폴더에 분리. 모든 정보를 한 번에 넣으면 AI가 오히려 혼란스러워합니다.
my-skill/ ├── SKILL.md ← 핵심 지침만 ├── references/api.md ← 상세 API └── assets/template.md ← 출력 템플릿
🎯
③ 너무 세세하게 지시하지 말 것
핵심만 알려주고 상황 판단은 AI에게 맡겨야 합니다. 모든 경우의 수를 통제하려 하면 오히려 유연성이 떨어집니다.
📝
④ Description 필드는 AI를 위해 쓸 것
"이 스킬이 뭘 하는가"가 아니라 "언제 이 스킬을 실행할지"로 써야 합니다.

❌ "코드 리뷰를 수행합니다" (기능 설명)
✅ "사용자가 'PR 리뷰해줘', '코드 검토', '보안 점검'을 요청할 때 실행" (트리거 설명)
핵심 통찰
하나의 카테고리만
좋은 스킬은 하나의 카테고리에 명확히 속합니다. "이 스킬이 뭘 하는 거지?"라는 질문이 나오면 범위가 너무 넓다는 신호.
몇 줄로 시작해도 됩니다
완벽한 스킬을 설계하려 하기보다 시작하고 반복하는 것이 핵심. 처음엔 몇 줄 메모 + 함정 하나로 시작.
실패할 때마다 한 줄
AI가 새로운 실수를 할 때마다 함정 목록에 한 줄 추가. 이것이 스킬을 점점 좋게 만드는 유일한 방법.

AI 시대의 핵심 읽을거리

에이전틱 엔지니어링, 하네스, Claude Code 내부 구조까지. 교육자와 실무자를 위한 엄선된 인사이트 모음.

🖥️ Claude Code, 프로그래밍을 넘어 더 많은 것으로 HOT
📌 핵심 메시지: Claude Code가 코딩 에이전트 → 작업 에이전트로 확장되고 있습니다. 데스크탑 앱 리디자인, Opus 4.7 디자인 도구, 병렬 세션이라는 세 가지 신호가 이런 변화를 보여줍니다. 디자인 도구 시장 전반에도 영향을 줄 수 있는 흐름으로 평가됩니다.
신호 1
🎨 HTML 프리뷰 패널 — 디자인 도구의 포석
코딩 도구에 왜 웹페이지 프리뷰가 필요했을까요?

Opus 4.7 디자인 도구가 이번 주 출시 예정입니다. 프롬프트로 웹사이트와 프레젠테이션을 만드는 자연어 디자인 도구예요. 데스크탑 앱에 HTML/PDF 프리뷰 패널을 먼저 넣은 이유가 여기 있습니다.

디자인을 만들면서 바로 확인할 수 있는 환경을 먼저 깔아놓은 겁니다. artifact로 랜딩 페이지를 만들 때보다, 전용 도구가 붙으면 비개발자도 결과물 수준이 완전히 달라질 것입니다.
신호 2
🧩 패널 조립형 레이아웃 — 결국 IDE가 정답
터미널, 파일 에디터, diff viewer, 프리뷰가 각각 독립 패널로 존재합니다. Drag-and-drop으로 원하는 위치에 배치. VS Code처럼 터미널은 아래, 에디터는 가운데, 파일 트리는 사이드바에 두는 방식입니다.

AI 코딩 에이전트를 쓸 때 가장 많이 보는 화면은 여전히 터미널이에요. 에이전트가 어떤 명령을 실행하는지 로그를 직접 봐야 신뢰가 생깁니다.

Anthropic은 터미널을 없애지 않고 앱 안에 그대로 품었고, 기존 CLI 플러그인과 hooks 설정도 그대로 작동합니다.
  • diff viewer 패널에서 라인별 코멘트 후 일괄 제출 가능
  • 파일 에디터 내장 — 에이전트 작업 중 수동 수정도 같은 창에서 처리
신호 3
🔀 병렬 세션 — Codex가 연 시장에 다른 답
Codex가 2월에 데스크탑 앱을 내놓으면서 병렬 에이전트 관리라는 카테고리를 열었습니다. 터미널 하나에 세션 하나로 쓰던 시절의 가장 아쉬웠던 부분.

이번 리디자인으로 하나의 창에서 여러 세션을 나란히 띄우고 사이드바에서 전환합니다. Git worktree 자동 격리까지 들어가서 에이전트끼리 코드 충돌이 없어요.

중요한 건 병렬로 돌리는 세션이 전부 코딩일 필요가 없다는 점입니다. 한쪽에서 버그 수정, 다른 쪽에서 랜딩 페이지 제작, 또 다른 쪽에서 프레젠테이션 생성이 동시에 가능합니다.
  • Codex 앱: 출시 3개월 만에 주간 활성 사용자 200만 돌파
  • Claude Code: worktree 격리로 세션 간 코드 충돌 방지
  • Routines 기능 같은 날 발표 — 반복 작업 스케줄링 지원
결론
🎯 세 요소가 가리키는 방향
공통점이 있습니다. Claude Code가 코딩 에이전트에서 작업 에이전트로 확장되고 있다는 겁니다.

① HTML 프리뷰 패널 → 디자인 도구의 출력 화면
② IDE형 레이아웃 → 코딩과 디자인을 같은 공간에서
③ 병렬 세션 → 여러 종류 작업을 동시에 돌리는 기반

모델을 더 똑똑하게 만드는 것만으로는 부족합니다. 그 모델이 할 수 있는 일의 범위를 넓히고, 그걸 담는 작업 환경까지 먼저 깔아야 합니다.
💡
시장의 반응: 디자인 도구 패러다임의 변화 신호
이번 발표 직후 디자인 도구 업계 주가에도 약간의 변동이 있었는데, 이는 "코딩 도구가 더 이상 코딩만 하지 않는다"는 신호로 받아들여졌기 때문입니다.

이미 많은 사용자가 소프트웨어 개발을 넘어 다양한 업무에서 AI 도구를 함께 쓰고 있고, 도구의 진화 속도도 빨라지고 있습니다. 결국 "도구보다는 무엇을 만들고 싶은가"라는 본질적인 질문이 점점 더 중요해지는 흐름입니다.

2. 에이전틱 엔지니어링 5가지 핵심 원칙
📌 AI 에이전트가 목표를 자율적으로 추구하는 시스템을 설계·구축·운영하는 기술. 핵심은 "위임(Delegate) → 검토(Review) → 소유(Own)" 모델입니다.
원칙 1
Planning (계획 수립)
즉시 실행 대신 복잡한 작업을 분해하고, 의존성을 파악하며, 실행 로드맵을 생성합니다. "생각 먼저, 행동 나중"의 원칙.
원칙 2
Tool Calling (도구 호출)
에이전트가 외부 시스템, API, 데이터베이스, 서비스와 상호작용할 수 있게 합니다. MCP, Skills가 여기에 해당.
원칙 3
Memory (기억 관리)
시맨틱: 사실적 지식 / 에피소드: 과거 경험 / 절차적: 방법론
단기(대화) + 장기(세션 간) 범위로 운영.
원칙 4
Human Oversight (인간 감독)
인간의 감독, 시스템 설계 문해력, 고판단 의사결정은 AI가 아무리 발전해도 핵심 역량으로 남습니다.
원칙 5
위임-검토-소유 (DRO) 모델
Delegate: AI에게 작업 위임
Review: 결과물 검토 및 피드백
Own: 최종 책임은 인간이 소유
선도 팀들이 수렴하는 운영 모델.
적용법
기획자-생성자-평가자 루프
Planner → Generator → Evaluator
이 피드백 루프가 멀티 시간 자율 코딩 세션 동안 반복되며 풀스택 앱을 만들어냅니다.

📦 NPM 배포 과정의 실수로 소스맵(.map) 파일을 통해 약 2,300개 파일·50만 줄 분량의 TypeScript 코드가 공개되어, 커뮤니티에서 이를 분석한 글이 다수 공유되었습니다. 본 페이지는 그 분석 글을 바탕으로 한 설계 인사이트 정리입니다.

Buddy System (다마고치)

발견 기능 2
Kairos 데몬 모드
항상 켜져서 백그라운드에서 기억을 통합하고 자율적으로 동작하는 선제적 AI 행동 시스템. Auto-Dream 자동 메모리 관리 기능 포함.
발견 기능 3
토큰 최적화 메커니즘
토큰 사용량이 임계값(유효 윈도우 - 13,000 토큰)을 넘으면 자동 압축 작동. Coordinator 팀 오케스트레이션 등 44개 미출시 기능 발견.

5. 하네스의 5가지 핵심 구성 요소
📌 LLM은 기본적으로 입력→텍스트 출력 함수일 뿐. 세션 간 상태 유지, 코드 실행, 실시간 정보 접근, 환경 구성은 모두 하네스가 해결해야 합니다.
구성 1
📁 파일시스템
작업 공간 제공, 진행 상황 기록, Git을 통한 버전 관리. 에이전트의 "작업 책상".
구성 2
🔒 샌드박스 환경
격리된 코드 실행 공간. 실수해도 프로덕션에 영향 없도록 안전한 실험실 제공.
구성 3
🧠 컨텍스트 관리
컨텍스트 부패 방지, 필요한 정보만 노출. CLAUDE.md, @import, .claude/rules/ 활용.
구성 4
🤖 서브 에이전트
중간 과정의 노이즈 흡수. 메인 에이전트의 컨텍스트를 보호하는 방패 역할.
구성 5
⚡ 백프레셔 메커니즘
자동 테스트, 타입 체크로 에이전트의 자기검증 강화. 실패 → 구조적 방지 장치 추가.
핵심 원칙
🎯 실패에서 학습하라
에이전트가 실패할 때마다 구조적으로 방지하는 장치를 추가. 최소한의 정보만 제공하고 한 번에 하나의 기능만 처리.
6. Claude Code 토큰 절약 — 무료 도구 10선
📌 대상: AI 코딩 도구를 자주 쓰는 개발자. 비개발자라면 "이런 흐름이 있구나" 정도로만 가볍게 봐주셔도 충분합니다.
💡 Claude Code를 본격적으로 쓰다 보면 토큰 리밋이 가장 자주 마주치는 병목입니다. 같은 작업을 더 적은 토큰으로 수행하도록 도와주는 오픈소스 도구들을 모았습니다. 모두 GitHub 무료 공개 프로젝트입니다.
01 · CLI 프록시
RTK ↗
터미널 출력을 컨텍스트에 넣기 전에 필터링하는 CLI 프록시. 바이너리 하나, 의존성 없음.
02 · 플러그인
Context Mode ↗
Playwright·GitHub·로그 등 raw 출력을 SQLite에 샌드박싱해서 깨끗한 요약만 대화에 넣어줍니다. 컨텍스트 98% 절감이라는 보고가 있습니다.
03 · MCP 서버
code-review-graph ↗
Tree-sitter로 코드베이스를 로컬 지식 그래프로 매핑해서 필요한 부분만 읽게 해줍니다. 대형 모노레포에서 49배 토큰 절감 사례.
04 · MCP 서버
Token Savior ↗
파일 전체 대신 심볼 단위로 코드를 탐색하는 MCP 서버. 69개 도구에 세션 간 메모리도 유지. 약 97% 절감이라는 수치를 제시.
05 · 스킬
Caveman Claude ↗
Claude를 원시인처럼 말하게 해서 출력 토큰을 65~75% 깎아주는 스킬. 한 줄 설치로 적용 가능하며, 결과물 정확도는 유지된다는 평.
06 · 드롭인 설정
claude-token-efficient ↗
CLAUDE.md 파일 하나로 응답을 간결하게 잡아주는 드롭인 설정. 코드 변경 없이 바로 적용.
07 · MCP 서버
token-optimizer-mcp ↗
캐싱·압축·스마트 도구 인텔리전스를 결합한 MCP 서버. 반복 출력을 압축해서 95% 이상 절감을 노립니다.
08 · 셋업 모음
claude-token-optimizer ↗
어떤 프로젝트든 5분이면 적용 가능한 셋업 프롬프트 모음. 문서 토큰이 11K → 1.3K 수준으로 줄어든 사례 보고.
09 · 분석 도구
token-optimizer ↗
컨텍스트를 조용히 잡아먹는 "고스트 토큰"을 찾아냅니다. 컴팩션 후에도 품질이 유지되도록 도와줍니다.
10 · 코드 검색 MCP
claude-context (by Zilliz) ↗
BM25 + 벡터 하이브리드 검색으로 코드베이스 전체를 컨텍스트로 만들어주는 도구. 검색 품질을 유지하면서 약 40% 절감.
🎯
10개를 다 설치할 필요는 없어요 — 워크플로에 맞게
· 터미널 출력이 많다면 → RTK
· 큰 코드베이스라면 → code-review-graph + Token Savior
· MCP 서버를 여러 개 쓴다면 → Context Mode
· 당장 토큰 리밋을 줄이고 싶다면 → Caveman Claude + claude-token-efficient

※ 각 프로젝트의 절감 수치는 제작자가 자체 측정한 값으로, 실제 환경에서는 차이가 있을 수 있습니다. 도입 전에 작은 프로젝트에 먼저 적용해보시길 권장합니다.

AI 한 명이 다 하는 게 아니라 기획자·생성자·평가자 3인조가 역할을 나눕니다. 이걸 아는 순간 결과가 완전히 달라집니다.

왜 단일 에이전트는 실패하는가 — 2가지 치명적 원인
문제 ①
문맥 불안 (Context Anxiety)
대화창이 꽉 찼다고 느끼면 작업을 대충 마무리 지으려는 현상. 긴 코딩 작업을 통째로 맡길 때 주로 발생합니다.

자신이 짠 엉망인 코드나 디자인에도 무조건 "훌륭하다"고 칭찬하는 현상. 특히 정답이 없는 디자인 영역에서 심각합니다.

🔧 Claude Code 혼자 앱 만들게 하는 하네스 설계 4단계
💬
왜 사람이 계속 개입하게 되는가
기능 하나가 끝날 때마다 다음 지시를 내려야 하고, 컨텍스트가 쌓이다 세션이 종료되면 이전 설계를 온전히 이어받지 못합니다. 엔지니어가 인수인계 없이 일을 시작하는 것과 같습니다. Anthropic은 이 문제의 원인이 모델 성능이 아닌 구조에 있다고 봤습니다.
1
플래너에게 스펙을 맡겨라
한두 문장 프롬프트를 주면 플래너 에이전트(Planner Agent)가 전체 제품 스펙으로 확장합니다. 초반에 구현 디테일을 너무 구체적으로 잡으면 그 오류가 이후 작업 전체로 번지기 때문에, "무엇을 만들지"에만 집중하도록 설계하는 게 핵심입니다.
2
인수인계 구조를 만들어라 — 파일 3개
초기화 에이전트(Initializer Agent)가 첫 세션에서 파일 3개를 만듭니다:

feature_list.json # 기능 목록 claude-progress.txt # 진행 상황 기록 init.sh # 서버 시작 방법 정의
이후 코딩 에이전트는 새 세션 시작 때마다 이 파일들을 읽고 어디서부터 이어갈지 파악합니다.
3
세션 시작 루틴을 고정하라
매 세션은 항상 같은 순서로 시작합니다:
① 진행 파일 확인 ② 기능 목록 읽기 ③ 테스트 실행 ④ 기능 하나 선택 → 구현 ⑤ 커밋 ⑥ 진행 파일 업데이트 ⑦ 다음 기능으로 → ①로 반복
이 사이클을 반복하면 사람 개입 없이 연속 작업이 가능합니다.
4
만드는 에이전트와 검토하는 에이전트를 나눠라
코드를 짜는 에이전트와 검토하는 에이전트를 분리하면 품질이 올라갑니다. 사람으로 치면 개발자와 QA를 나누는 것과 같습니다. 정답이 없는 프론트엔드 디자인 영역에서도 효과적이었다고 합니다.
🙌 하네스 설계는 방법을 안다고 바로 되는 건 아닙니다. 구조를 이해하고 직접 써보면서 자기 프로젝트에 맞게 깎아나가야 합니다. 그래도 오피셜한 기준이 있으니 시작은 해볼 수 있습니다.

🔁 2026 AI 핵심 패턴 · Loop

복잡한 파이프라인보다, 잘 설계된 반복 한 줄

자동 반복 실험을 활용하면, 시간을 들이지 않고도 의미 있는 결과를 꾸준히 쌓을 수 있습니다.

4가지 핵심 루프 패턴
① Ralph Loop
bash 루프 기반 에이전트
Step 1. while :; do ... done 형태의 단순한 bash 루프를 짭니다.

Step 2. PRD에 passes: true 등 명확한 테스트 통과 조건을 설정합니다.

Step 3. 매 회차마다 컨텍스트 윈도우를 깨끗하게 비워 성능 저하를 막습니다.

Step 4. 에이전트가 실패하면 자동으로 재시도하도록 두고, 결과만 확인합니다.

단순한 bash 한 줄도 검증 조건만 명확하면 충분히 강력합니다.
② RLM (재귀 추론)
모델이 스스로 맥락을 결정
Step 1. 긴 문서를 LLM에 직접 넣지 않고 Python REPL 환경에 올립니다.

Step 2. 모델이 직접 코드를 짜서 변수를 쪼개고 탐색해 만듭니다.

Step 3. 파악한 맥락 중 필요한 부분만 골라 자기 자신을 재귀 호출합니다.

Step 4. 요약 손실 없이 100배 넘는 컨텍스트를 완벽하게 소화합니다.

모델 스스로 맥락을 결정합니다. 기존 RAG와는 근본적으로 다릅니다.
③ autoresearch
자동으로 100회 반복하는 무인 실험
Step 1. AI에게 train.py와 목표 방향성만 적어줍니다.

Step 2. 정확히 5분간 훈련을 돌려 성능이 오르면 Commit, 아니면 Reset합니다.

Step 3. 사람이 일하지 않는 시간 동안 이 무인 실험을 100회 이상 자동 반복합니다.

Step 4. 다음 날 어떤 변경이 효과적이었는지 로그를 확인합니다.

한 사람의 시도가 아닌, 연구 커뮤니티 규모의 실험을 자동으로 흉내낼 수 있습니다.
④ 반복의 힘
자동 반복으로 쌓이는 개선
Step 1. 모델 크기 대신 추론 시점의 연산(Test-time compute)을 높입니다.

Step 2. 객관적이고 검증 조건이 명확한 테스트 루프를 설정합니다.

Step 3. 작업이 자동으로 이어지도록 충분한 API 토큰 예산을 배정합니다.

Step 4. 코드를 넘어서 기존에 없던 새로운 결과물을 자동으로 만들어 봅니다.

자리에 없는 시간 동안에도 작은 개선이 자동으로 누적되도록 설계할 수 있습니다.
왜 Loop가 승부처인가
🌙 수면 시간 = 개발 시간
자는 8시간 동안 루프가 돌면 아침에 100번의 실험 결과가 쌓입니다. 복잡한 파이프라인 없이도 압도적인 속도.
💰 비용보다 반복
비싼 모델·복잡한 세팅보다 명확한 조건 + 단순한 루프가 더 강합니다. 검증 조건이 핵심.
🎯 조건 명확성이 핵심
루프의 성패는 "무엇을 성공으로 볼 것인가"를 얼마나 명확히 정의했느냐에 달려 있습니다.
